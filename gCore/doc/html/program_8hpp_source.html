21 /*
22  * Okay, forget GLSW. We'll come up with our own, The High Level
23  * Shading Preprocessor, HLSPrep.
24  *
25  * Sections of a shader source file are cut up and labeled with the
26  * syntax " @<directive> '<name>(.<name>)*' ". This makes any file
27  * that uses this system uncompilable on its own, which is a GOOD
28  * thing. No one can screw it up, though it does mean you need
29  * this system.
30  *
31  * The directives are:
32  *
33  * block - a named block of source code
34  * vert - vertex shader source follows
35  * frag - fragment shader source follows
36  * geo - geometry shader source follows -- There are actually two kinds...
37  *
38  * end - end the current block
39  *
40  * insert - insert the contents of the given block into this block at this point
41  * ref - the file makes reference to another file's contents
42  * using - indicates what version of GLSL the current block uses.
43  *
44  * The directives block, vert, frag, and geo all define a path for a block of
45  * source code. The block comprises all the text between the line after the
46  * directive and the line before the next blocking directive or EOF.
47  *
48  *
49  * "block" gives a name to the source, but does not flag that source as a
50  * compilable shader stage. Anything can appear in a block. "vert", "frag", and
51  * "geo" define a path and flag a source block as a compilable shader stage.
52  * The shader blocks must meet certain requirements to be "valid" (I have not
53  * yet defined these, but they'll be the minimum requirements of the respective
54  * shader stage).
55  *
56  * The "insert" directive substitutes the source associated with the given path
57  * at the location of the directive.
58  *
59  * The "ref" directive indicates that the file makes use of paths from other
60  * source files. The path given indicates what paths it will be using;
61  * the same matching rules apply here as well.
62  *
63  * Finally, the "using" directive declares what version the current block is using.
64  * GLSL's preprocessor does provide it's own directive for this purpose, the
65  * "#version" directive, and you are free to use it instead. However, the GLSL
66  * directive has an important syntactical restriction: it must appear in shader
67  * source before any other source except whitespace and comments. gEngines
68  * annotations are intended to make adaptive shader code easier, and part of that
69  * is providing a way to dynamically pick the shader source that fits the available
70  * version of GLSL, assembling a functioning shader from parts written for different
71  * versions.
72  *
73  * Because the blocking system is swapping source around and picking the code
74  * that fits the hardware, it must provide it's own facility for the coder to
75  * label the version of GLSL they intend to use. Since gEngine's directives
76  * operate on top of the GLSL compiler and preprocessor, having it manipulate
77  * the "#version" directive instead would violate the expectations of the coder.
78  *
79  * So, either only use "#version" and do not use the dynamic code generation
80  * of gEngine or only use "@using". Combining them WILL, I repeat, WILL lead
81  * to strange bugs, mostly in the form of compiler errors.
82  *
83  * -- 2017 me: I suspect that the thing to do would be to make the @using
84  * directive define a variant of a block specialized for a particular version
85  * of GLSL, letting gEngine decide what the appropriate version to use is
86  * based ont he available hardware/driver support. This also means it requires
87  * it's own terminating directive.
88  *
89  * -- Handle Matching --
90  *
91  * Fully qualified names are always unique, so each named block has a unique
92  * path. Names are broken into tokens by the period (.) character, and this
93  * is used to dynamically match paths. Each token represents the 'order' of
94  * the match. For instance, the string "HandleA" matches both the paths
95  * "HandleA.foo" and "HandleA.bar", but "HandleA.f" does not match "HandleA.foo".
96  *
97  *
98  * A block ends at an end directive or an EOF. The current block can be extended
99  * with a new token by using "..." before it. For example:
100  *
101  * @vert "Vertex"
102  *
103  * @begin "...LowDef"
104  * @end
105  *
106  * @begin "...HighDef"
107  * @end
108  *
109  * @end
110  *
111  * Thus creating two paths, "Vertex.LowDef" and "Vertex.HighDef". Because
112  * of the potential ambiguity, the "insert" directive and other mutagenic
113  * directives cannot use the "..." syntax (what would it mean, anyway?) and
114  * instead must use a name that starts at the root of the paths.
115  *
116  *
117  * @begin 'ExtHeader'
118  *
119  * @begin '...v25'
120  * // ...
121  * @end
122  *
123  * @begin '...v31'
124  * // ...
125  * @end
126  *
127  * @end
128  *
129  * @begin 'SphHarmonics'
130  * // ...
131  * @end
132  *
133  * @vert 'Vertex'
134  *
135  * // Code and stuff
136  * @ifversion 250
137  *
138  * @insert 'ExtHeader.v25'
139  * @insert 'SphHarmonics'
140  *
141  * // ...
142  * @endversion
143  *
144  * @ifversion 310
145  *
146  * @insert 'ExtHeader.v31'
147  * @insert 'SphHarmonics'
148  *
149  * // ...
150  * @endversion
151  *
152  * @end
153  *
154  * <code-block> ::= what's left over, basically
155  * <directive> ::= "@" <directive-expression> <EOL>
156  * <directive-expression> ::= <block-expression> | <markup-expression>
157  * <block-expression> ::= <block-tag> "'" <path-expression> "'"
158  * <block-tag> ::= "begin" | "vert" | "frag" | "geo"
159  * <path-expression> ::= <path> | "..." <path>
160  * <path> ::= <path-tag> | <path-tag> "." <path>
161  * <path-tag> ::= <path-character> | <path-character> <path-tag>
162  * <path-character> ::= <letter> | <number> | "_"
163  * <markup-expression> ::= <terminator-expression> | <version-expresion> | <version-end-expression> | <insertion-expression>
164  * <terminator-espression> ::= "end"
165  * <version-expression> ::= "ifversion" <version-tag>
166  * <end_version-expression> ::= "endversion"
167  * <version-tag> ::= <number> <number> <number>
168  * <insertion-expression> ::= "insert" "'" <path-tag> "'"
169  *
170  *
171  * */
172 
173 /*
174  * A Shader object manages a single shader source file. We adopt
175  * the syntax and behavior of the GL Shader Wrangler, which is
176  * straight forward to replicate and undoubtedly useful. Using
177  * the GLSW library itself was not done because that library is
178  * C-compatible and requires the inclusion of another string library.
179  * C++'s native <string> library is sufficient for the task.
180  *
181  * This implementation must duplicate the interpretive behavior of
182  * the GLSW. However, since GLSW is a centralized service that
183  * coordinates loading and accessing shader effect files, we may be
184  * at odds.
185  *
186  * Each Shader object knows the file it manages; it reads this file
187  * and stores the individual pieces of the shader according to the
188  * Key specified in the file for each piece. Upon request, the
189  * appropriate shader source is assembled from the pieces.
190  * */
